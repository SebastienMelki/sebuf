Create `proto/sebuf/krakend/krakend.proto` with the following content:

```protobuf
syntax = "proto3";
package sebuf.krakend;

import "google/protobuf/descriptor.proto";

option go_package = "github.com/SebastienMelki/sebuf/krakend;krakend";

// GatewayConfig sets service-wide KrakenD defaults.
// Applied to a service via (sebuf.krakend.gateway_config).
message GatewayConfig {
  // Backend host URLs (e.g., "http://backend:8080"). Required.
  // Must include scheme (http:// or https://).
  // Multiple hosts enable KrakenD load balancing.
  repeated string host = 1;

  // Default timeout for all endpoints in this service.
  // KrakenD duration format: "2s", "500ms", "1m".
  // Omit to use KrakenD's built-in default.
  string timeout = 2;
}

// EndpointConfig overrides service-level defaults for a specific RPC.
// Applied to an rpc method via (sebuf.krakend.endpoint_config).
message EndpointConfig {
  // Override backend host for this specific endpoint.
  repeated string host = 1;

  // Override timeout for this specific endpoint.
  string timeout = 2;
}

extend google.protobuf.ServiceOptions {
  GatewayConfig gateway_config = 51001;
}

extend google.protobuf.MethodOptions {
  EndpointConfig endpoint_config = 51002;
}
```

Then generate Go code. Update the `proto` target in `Makefile` to include the krakend proto:

Add to the `proto` target's `--go_opt` flags:
```
--go_opt=Msebuf/krakend/krakend.proto=github.com/SebastienMelki/sebuf/krakend
```

And add to the proto file list:
```
proto/sebuf/krakend/krakend.proto
```

Then run `make proto` to generate `krakend/krakend.pb.go`.

Run `go mod tidy` afterward.

IMPORTANT: The extension numbers 51001 and 51002 are chosen to be well above the existing sebuf.http range (50003-50020) per the research document. The `repeated string host` field supports KrakenD's multi-host load balancing.

**Create `internal/krakendgen/types.go`** with typed Go structs mirroring KrakenD's endpoint/backend JSON schema:

```go
package krakendgen

// Endpoint represents a single KrakenD endpoint configuration.
type Endpoint struct {
    Endpoint          string    `json:"endpoint"`
    Method            string    `json:"method"`
    OutputEncoding    string    `json:"output_encoding"`
    Timeout           string    `json:"timeout,omitempty"`
    InputHeaders      []string  `json:"input_headers,omitempty"`
    InputQueryStrings []string  `json:"input_query_strings,omitempty"`
    Backend           []Backend `json:"backend"`
}

// Backend represents a KrakenD backend configuration.
type Backend struct {
    URLPattern string   `json:"url_pattern"`
    Host       []string `json:"host"`
    Method     string   `json:"method"`
    Encoding   string   `json:"encoding"`
}
```

Fields use `omitempty` for `timeout`, `input_headers`, and `input_query_strings` per the research: when no annotations exist, these fields should be absent from JSON (KrakenD zero-trust model treats absent as "forward nothing").

**Create `cmd/protoc-gen-krakend/main.go`** following the protoc-gen-openapiv3 pattern exactly:

- `readRequest()` — read CodeGeneratorRequest from stdin
- `createPlugin()` — create protogen.Plugin
- `generateFiles()` — iterate plugin.Files, skip non-Generate files, iterate services
- `writeResponse()` — write CodeGeneratorResponse with FEATURE_PROTO3_OPTIONAL

For now, the generator should iterate services and produce a minimal placeholder output: an empty JSON array `[]` written to `{ServiceName}.krakend.json`. This ensures the plugin binary is functional and produces output that protoc accepts.

The entry point should NOT parse any plugin parameters (per user decision: no `--krakend_opt=host=...` flag; host lives in annotations).

Import `internal/krakendgen` to establish the package dependency even though types aren't used yet in this task (they will be in Plan 02).

After creating the files, run `rm -rf bin && make build` to verify the new binary builds. The Makefile auto-discovers `cmd/protoc-gen-krakend/`.
