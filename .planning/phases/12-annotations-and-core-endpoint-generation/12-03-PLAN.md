Add auto-derived header and query string forwarding to the KrakenD endpoint generator by reading existing sebuf.http header and query annotations.

Purpose: KrakenD's zero-trust model blocks all parameter forwarding by default. Without explicit `input_headers` and `input_query_strings`, authentication headers and query filters break silently through the gateway.
Output: Generator produces endpoints with correct forwarding lists derived from proto annotations.
</objective>

<task type="auto">
  <name>Task 1: Add header forwarding from service and method header annotations</name>
  <files>
    internal/krakendgen/generator.go
  </files>
  <action>
Update the `GenerateService` function in `internal/krakendgen/generator.go` to populate `InputHeaders` on each endpoint.

For each RPC, after building the endpoint:

1. Call `annotations.GetServiceHeaders(service)` to get service-level headers.
2. Call `annotations.GetMethodHeaders(method)` to get method-level headers.
3. Call `annotations.CombineHeaders(serviceHeaders, methodHeaders)` to merge them (method overrides service for same-name headers -- this is the existing merge semantic from the annotations package).
4. Extract header names from the combined list: `h.GetName()` for each header.
5. Set `endpoint.InputHeaders = headerNames` if the list is non-empty.
6. If no headers are annotated, leave `InputHeaders` as nil -- the `omitempty` JSON tag will omit the field entirely (FWD-03: never emit empty arrays).

Create an unexported helper function:
```go
func deriveInputHeaders(service *protogen.Service, method *protogen.Method) []string
```

This returns nil if no headers are annotated, or a sorted list of header names otherwise. Sort the list for deterministic output (important for golden file tests).

IMPORTANT: The `CombineHeaders` function already handles the merge semantics (method overrides service). Do NOT re-implement merge logic -- just reuse it.
  </action>

  </verify>
  <done>
